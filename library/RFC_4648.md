NORMALISED EXTRACT
Alphabets (exact):
- Base64 (standard): ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/
- Base64url (URL and filename safe): ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789-_
- Base32 (RFC 4648): ABCDEFGHIJKLMNOPQRSTUVWXYZ234567
- Base16 (hex, RFC style): 0123456789ABCDEF

Bits per character:
- Base64: 6 bits/char
- Base32: 5 bits/char
- Base16: 4 bits/char

Encoding algorithm (step sequence — implementer-level):
1. Treat input as a stream of octets (bytes). Use Uint8Array as input type.
2. Group input into 24-bit blocks (three octets). For each 24-bit block:
   a. Split the 24 bits into four 6-bit values (most-significant-first order).
   b. Map each 6-bit value v (0..63) to the Base64 alphabet character at index v.
3. If the final block has fewer than 3 bytes, pad the input with zero bytes to make a full 24-bit block, perform the same mapping to produce 4 characters, then replace the last 1 or 2 output characters with '=' padding characters so that the total encoded length is a multiple of 4 (standard Base64 padding rule).
4. For Base64url, use the Base64url alphabet (see above) and optionally omit padding '=' if using the "base64url without padding" variant.

Decoding algorithm (step sequence — implementer-level):
1. Normalize the input string: remove whitespace and line breaks; for Base64url accept '-' and '_' in place of '+' and '/'.
2. Replace any padding characters '=' with zero-value bits when reconstructing; count padding to drop corresponding output octets at the end.
3. Map each encoded character to its numeric value (0..63 for Base64). Concatenate the 6-bit values into a stream of bits, then re-slice the bit stream into 8-bit bytes (most-significant-first) and output as Uint8Array.
4. Validate illegal characters and incorrect padding lengths and throw an error or return a decode failure if encountered.

URL-safe variant and padding notes:
- Base64url uses '-' and '_' instead of '+' and '/'.
- Some applications require removal of padding ('=') — this changes the canonical representation and must be documented and reversible.

SUPPLEMENTARY DETAILS
- Use bit-buffer operations to avoid big-integer arithmetic: maintain an integer accumulator and a bit-count, push bytes into accumulator (acc = (acc << 8) | byte; bits += 8). While bits >= 6, emit alphabet[(acc >> (bits - 6)) & 0x3F]; bits -= 6. After processing all input, if bits > 0 emit alphabet[(acc << (6 - bits)) & 0x3F] and, for standard Base64, pad output with '=' to multiple of 4.
- For decoding: similar accumulator approach: for each input char append its 6-bit value into acc (acc = (acc << 6) | value; bits += 6). While bits >= 8, emit (acc >> (bits - 8)) & 0xFF; bits -= 8.
- Implementation must accept and produce Uint8Array; when interfacing with Node Buffer, convert buffer <-> Uint8Array without copying when possible (Buffer.from(Uint8Array) and Uint8Array.from(Buffer) patterns).

REFERENCE DETAILS — API SIGNATURES (recommended for this project)
- encode(encodingName: string, data: Uint8Array): string
  - encodingName examples: "base64", "base64url", "base32", "base16", or a custom encoding registered by createEncoding
  - returns canonical encoded string (variant-specific).
- decode(encodingName: string, input: string): Uint8Array
  - reverse of encode, must validate input and throw on invalid characters.
- createEncoding(name: string, charset: string): { name, charset, bitsPerChar, encode(data: Uint8Array): string, decode(s: string): Uint8Array }
  - charset must be a string of unique printable characters; bitsPerChar = log2(charset.length).
- listEncodings(): Array<{ name: string, bitsPerChar: number, charsetSize: number }>
